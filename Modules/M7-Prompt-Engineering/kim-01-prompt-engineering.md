## Вариант задания

Продолжайте предметную область, назначенную преподавателем в лабораторных №1–14. Основную сущность возьмите из своего варианта №1, а сценарий внешнего API выберите применительно к той же области. Отдельного варианта для этой работы нет.

## Порядок выполнения

Перед началом изучите [инструкцию по промптам](../../resources/llm-prompts/prompt-chains.md). Технические формулировки, API, signatures и diagnostics передавайте на английском.

1. **Самостоятельно.** Выберите сущность своего варианта с не менее чем шестью разнотипными полями и определите проверяемые требования к Java-модели.
2. **Самостоятельно.** Составьте структурированную постановку задачи. В ней явно задайте:
   - цель и фактический контекст проекта;
   - контекст предметной области;
   - точную задачу и перечень полей;
   - версию Java и допустимую форму модели;
   - требования к Jackson-аннотациям;
   - ограничения и запрещённые зависимости;
   - требуемый формат ответа;
   - критерии, по которым ответ будет принят.
3. **С ИИ-агентом.** Если используется чат, передайте минимально достаточные сигнатуры и зависимости. Если используется coding agent, укажите область поиска в репозитории и разрешите ему самостоятельно найти релевантные файлы. Получите только Java-модель в требуемом формате.
4. **Проверка результата.** Скомпилируйте модель в отдельном пакете, проверьте типы, Jackson-сериализацию, именование, изменяемость и ограничения; зафиксируйте исправления.
5. **Самостоятельно.** Выберите уместную операцию внешнего API и подготовьте пример запроса, ответа, ограничения и критерии проверки.
6. **С ИИ-агентом.** Передайте сценарий API, пример запроса/ответа, ограничения и критерии приёмки. Запросите архитектуру REST-клиента, модель ответа и стратегию обработки ошибок без реализации.
7. **Проверка результата.** Проверьте архитектуру, исправьте её и зафиксируйте утверждённую версию.
8. **С ИИ-агентом.** Передайте утверждённую архитектуру и запросите реализацию только модели ответа.
9. **Проверка результата.** Скомпилируйте и проверьте модель ответа; в следующий этап передавайте только подтверждённую версию.
10. **С ИИ-агентом.** Передайте архитектуру и проверенную модель, затем запросите реализацию клиента на стандартном `HttpClient`.
11. **Проверка результата.** Проведите code review всех полученных фрагментов:
   - компиляция и соответствие Java 17;
   - таймауты и обработка сетевых ошибок;
   - обработка кодов HTTP и некорректного тела ответа;
   - отсутствие секретов и небезопасного логирования;
   - корректная работа с прерываниями;
   - именование, зависимости и соответствие архитектуре.
12. **Проверка результата.** Сравните первоначальные и исправленные результаты, повторите сборку или тесты и укажите причину каждого изменения.
13. **Проверка результата.** Сохраните точные промпты, существенные ответы и связи этапов; убедитесь, что секреты, персональные данные и закрытый код не передавались.

Фиксированная персона вроде «ты — senior Java-разработчик» не является обязательной частью постановки. Она допустима только тогда, когда действительно задаёт полезную перспективу review; в остальных случаях важнее цель, доступный контекст, ограничения и проверяемые критерии готовности.

## Уровневые задания КРМ

Обязательная часть измеряет **Б** по LLM-5.1 и LLM-5.2. Задания С/П назначаются преподавателем заранее, выполняются после обязательной части и оцениваются по накопительной [модели измерения](../../methodical-guidelines/teachers-assessment/krm-level-model.md).

### C15-LLM5.1 — Выбор и адаптация шаблонов

1. Выберите не менее двух разных шаблонов промпта для одной задачи варианта и до запуска обоснуйте, какие свойства задачи определяют выбор.
2. Адаптируйте шаблоны под ограничения, формат и риски задачи; одной замены предметных существительных недостаточно.
3. Подготовьте минимум 10 тест-кейсов с эталонными проверяемыми свойствами, валидатор формата и повторяемый регрессионный прогон.
4. Сравните результаты на одинаковых входах и аргументированно выберите итоговый шаблон.

Полное выполнение подтверждает **С** по LLM-5.1 при достигнутом Б.

### C15-LLM5.2 — Условная цепочка

1. Реализуйте цепочку мыслей как последовательность явных проверяемых промежуточных артефактов, например `план → извлечённые факты → проект результата → независимая проверка`. Не запрашивайте и не оценивайте скрытые внутренние рассуждения модели.
2. Добавьте минимум одну явную ветвь: уточнение неполного входа, повтор после нарушения формата или независимую верификацию факта.
3. Зафиксируйте условие перехода машинно-проверяемым правилом; модель не должна сама объявлять свой ответ корректным.
4. Проверьте основную ветвь, каждую альтернативную ветвь и ограничение числа повторов.

Полное выполнение подтверждает **С** по LLM-5.2 при достигнутом Б.

### P15-LLM5.1 — Сложные шаблоны

1. Спроектируйте сложный шаблон с множеством отобранных примеров, включая обычные, граничные и контрпримеры.
2. Проведите ablation-сравнение минимум трёх конфигураций примеров на фиксированном тестовом наборе.
3. Проанализируйте точность, устойчивость формата и стоимость/длину контекста; обоснуйте финальный состав шаблона.

Полное выполнение подтверждает **П** по LLM-5.1 только после Б и С. П по LLM-5.2 измеряется отдельным заданием проекта №16, поскольку требует инструментального доступа к attention/logits.
