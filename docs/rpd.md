# Рабочая программа дисциплины

## 1. Общие сведения

| Поле | Значение |

## 2. Логика обновления курса

Курс сохраняет фундаментальное ядро ООП, но помещает его в современный инженерный контекст. Одна предметная область проходит через классы, коллекции, ввод-вывод, конкурентную обработку, JDBC и ETL. Каждая следующая работа использует проверенные результаты предыдущей. ИИ-агент применяется только для явно ограниченных артефактов; компиляция, запуск, анализ и окончательное решение остаются ответственностью студента.

## 3. Цель и задачи

**Цель:** сформировать системные навыки Java-разработки для создания надёжного, поддерживаемого серверного ПО и ETL-решений с осознанным применением ИИ-инструментов.

1. Освоить классы, инкапсуляцию, наследование, полиморфизм и интерфейсы.
2. Использовать коллекции, generics, исключения, файловый ввод-вывод и прикладные библиотеки Java.
3. Освоить основы многопоточности и работы с реляционными БД через JDBC.
4. Реализовать ETL-пайплайн с обработкой ошибок, логированием, метриками и идемпотентной загрузкой.
5. Составлять и проверять структурированные промпты и цепочки промптов, соблюдая границы применения ИИ-агента.

## 4. Пререквизиты

- переменные, типы, условия, циклы, функции и массивы;
- базовые алгоритмы и структуры данных;
- дискретная математика;
- основы реляционных БД, SQL DDL/DML, транзакций, ACID, индексов и нормализации.

## 5. Планируемые результаты

По завершении дисциплины обучающийся сможет:

- проектировать объектно-ориентированные модели и реализовывать их на Java;
- применять коллекции, generics, исключения и библиотеки структурированных данных;
- реализовывать и проверять конкурентную обработку данных;
- работать с PostgreSQL через JDBC: CRUD, транзакции, batch insert и UPSERT;
- собирать воспроизводимый ETL-пайплайн с диагностикой и идемпотентностью;
- составлять структурированные промпты и связанные цепочки;
- критически проверять созданные ИИ-агентом код, тесты и документацию.

## 6. Компетенции и профессиональный ролевой профиль КРМ v3.0

Основной ролевой профиль дисциплины — **Data Engineer (Инженер по данным)**, **MLOps (Специалист по эксплуатации ИИ)** и **TADS (Технический аналитик данных)**. Наименование роли и рекомендуемый уровень компетенции взяты из листа `Роли-компетенции наглядно` [КРМ v3.0](../data/krm-v3.0.xlsx). Дисциплина формирует часть компетенций этих ролей, но сама по себе не присваивает профессиональную роль.

| Компетенция | Профессиональная роль по КРМ v3.0 | Рекомендованный уровень в матрице «Роли–компетенции» | Вклад дисциплины |
|---|---|:---:|---|
| PL-2. Java/Scala/Kotlin | Data Engineer (Инженер по данным) | С | сбор, проверка, очистка и подготовка данных; ETL |
| PL-2. Java/Scala/Kotlin | MLOps (Специалист по эксплуатации ИИ) | С | JVM-решения в процессах жизненного цикла ИИ и работе с данными |
| PL-2. Java/Scala/Kotlin | TADS (Технический аналитик данных) | С | проектирование и разработка пайплайнов обработки и хранения данных |
| LLM-5. Промпт-инженерия | AI Architect (Архитектор ИИ) | Б | проектирование проверяемого взаимодействия с генеративной моделью |
| LLM-5. Промпт-инженерия | ML Engineer (Инженер МО) | С | дизайн промптов и проверка их эффективности |
| LLM-5. Промпт-инженерия | MLOps (Специалист по эксплуатации ИИ) | Б | воспроизводимое применение промптов в инженерном процессе |

Для PL-2 названные роли также указаны в трудовых функциях карточки компетенции; для LLM-5 карточка называет AI Architect и ML Engineer, а связь с MLOps следует непосредственно из матрицы. Обязательная траектория дисциплины подтверждает уровень **Б**. Уровни **С** и **П** фиксируются только по отдельным уровневым заданиям из ФОС; уровень С по PL-2 позволяет сопоставить результат студента с рекомендуемым уровнем основного ролевого профиля.

## 7. Тематический план

Элемент **7.1 «Инженерная работа с ИИ-ассистентами и coding agents»** проводится в начале курса, до лабораторной №1. За 2 академических часа рассматриваются различия чата и coding agent, структура проверяемой постановки задачи, контекст репозитория, `AGENTS.md`, разрешения на файлы и команды, локальные модели и MCP, безопасность и цикл «план → изменение → запуск → анализ → исправление». Такое расположение позволяет применять ИИ-инструменты осознанно во всех последующих работах.

**7.2. Проверяемая постановка задач ИИ, 2 а.ч.** Студент формулирует задачу без обязательной persona-конструкции: задаёт цель, фактический контекст, входные данные, ограничения, формат, критерии приёмки и способ проверки. Связанная последовательность для Java REST-клиента передаёт следующему этапу только проверенные архитектуру и модель; итог компилируется, тестируется и проходит code review.

**7.3. Командный мини-проект, 4 а.ч.** Команда из 2–3 человек выбирает один из вариантов своих участников и реализует [серверный ETL-сервис](../Project/README.md) для этой предметной области: воспроизводимые грязные CSV/JSON-данные, конкурентный ETL, PostgreSQL, идемпотентный UPSERT, DLQ, метрики и HTTP API на `JDK HttpServer`. Можно использовать стартовый [датасет](../resources/datasets/README.md), но предпочтительно подготовить и проверить собственный генератор. Работа ведётся на простой Kanban-доске и в Git; ИИ-агент используется для ограниченных задач, а предметная модель, архитектура и ядро обработки утверждаются командой.

Контактная тематическая часть включает лекционные и лабораторные занятия; оставшийся объём до 108 часов отводится на самостоятельную работу, проект и подготовку к экзамену. Детализация часов утверждается учебным планом реализации дисциплины.
