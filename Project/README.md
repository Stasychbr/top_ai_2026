## Цель

Собрать изученные средства Java в одно серверное приложение: получить грязные
данные выбранной предметной области из нескольких источников, выполнить
конкурентный ETL, сохранить очищенный результат в PostgreSQL и предоставить
доступ к импорту и данным через небольшой HTTP API. ИИ-агент используется для
ограниченных задач, а команда проверяет каждый полученный артефакт.

Перед началом изучите [предоставленные датасеты](../resources/datasets/README.md)
и инструкции по [генерации данных](../resources/llm-prompts/data-generation.md),
[цепочкам промптов](../resources/llm-prompts/prompt-chains.md),
[тестированию](../resources/llm-prompts/testing-and-debugging.md) и
[работе coding agent с репозиторием](../resources/llm-prompts/README.md).

## Контракт предметной модели

До реализации зафиксируйте таблицу соответствия:

| Исходный столбец | Поле Java | Столбец БД | Тип | Правило проверки и очистки |
|---|---|---|---|---|
| пример: `Код записи` | `businessKey` | `business_key` | `String` | обязательное уникальное значение |

Модель должна включать:

- устойчивый бизнес-ключ;
- основное наименование или идентификатор сущности;
- минимум два категориальных либо статусных поля;
- минимум одно числовое поле и одну дату;
- время создания или обновления записи;
- систему-источник;
- дополнительные поля, необходимые выбранной области.

Названия полей и таблиц должны отражать предметную область. Входные названия
можно преобразовать в английские Java-идентификаторы. Для каждого поля
зафиксируйте допустимость пропуска, формат и действие при ошибке: нормализация,
значение по умолчанию, пропуск поля либо отправка всей записи в DLQ.

Логический ключ определяется командой на основе варианта. При нескольких
корректных версиях сохраняется запись с наиболее поздним временем обновления.
Конфликтующие записи с одинаковым ключом и временем должны обрабатываться по
документированному детерминированному правилу.

## Обязательная архитектура

- Java 17 или новее, Maven либо Gradle;
- Jackson для JSON и PostgreSQL через JDBC;
- `ExecutorService` для параллельного чтения нескольких файлов;
- `JDK HttpServer` из модуля `jdk.httpserver`, без обязательного серверного
  фреймворка;
- конфигурация подключения к БД и каталога импорта вне исходного кода;
- локальные справочники для воспроизводимой нормализации категорий,
  сокращений и статусов;
- логирование этапов и DLQ-файл для отклонённых записей.

Допускается переиспользовать и доработать код лабораторных №7–14. Простое
копирование нескольких несвязанных проектов без единого запуска не считается
интеграцией.

## База данных

Минимальная схема содержит:

- основную таблицу выбранной сущности с уникальным бизнес-ключом;
- минимум одну связанную таблицу или таблицу-справочник, оправданную предметной
  областью;
- `import_runs` — идентификатор импорта, состояние, время начала и завершения и
  итоговые метрики.

Загрузка выполняется через `PreparedStatement` и UPSERT. Повторный импорт
одинаковых файлов не создаёт новых логических записей и не искажает метрики.

## HTTP API

Все ответы, кроме пустых технических ответов, имеют
`Content-Type: application/json; charset=utf-8`.

### `POST /imports`

Тело запроса:

```json
{
  "files": ["source-a.csv", "source-b.json"]
}
```

Разрешены только имена файлов внутри настроенного каталога `data/inbox`;
абсолютные пути и выход через `..` отклоняются. Сервер возвращает
`202 Accepted`:

```json
{
  "importId": "generated-id",
  "status": "QUEUED"
}
```

### `GET /imports/{id}`

Возвращает одно из состояний `QUEUED`, `RUNNING`, `COMPLETED`,
`COMPLETED_WITH_ERRORS`, `FAILED` и метрики `extracted`, `valid`, `rejected`,
`duplicates`, `inserted`, `updated`.

### `GET /records`

Возвращает очищенные записи выбранной предметной области. Поддерживаются
необязательные параметры `businessKey`, `status`, `limit` и минимум один
документированный фильтр предметной области. Некорректный `limit` приводит к
ответу `400`, а не к необработанному исключению.

### `GET /health`

Возвращает состояние приложения и доступности PostgreSQL. В ответе не должно
быть строки подключения, пароля или другой чувствительной конфигурации.

Аутентификация для учебного проекта не требуется. Неизвестный маршрут
возвращает `404`, неверный запрос — `400`, внутренняя ошибка — `500` с
безопасным сообщением и полной диагностикой только в журнале.

## ETL и тестовые данные

1. **Самостоятельно.** Зафиксируйте схему, бизнес-ключ, правила валидности и
   распределение тестовых случаев.
2. **Самостоятельно.** Выберите способ подготовки данных:
   - использовать предоставленный CSV как стартовый fixture и дополнить его;
   - подготовить набор полностью самостоятельно, если fixture отсутствует или
     команда хочет точнее отразить свой вариант.
3. **Проверка результата.** Обязательный набор должен содержать не менее трёх
   файлов суммарным объёмом не менее 1000 записей; среди файлов должны быть
   представлены и CSV, и JSON. Простое тиражирование одинаковых строк не
   засчитывается.
4. **Самостоятельно.** Включите межфайловые дубликаты, конфликтующие версии и
   минимум пять классов проблем: пропуски обязательных значений, неверные
   числа, некорректные даты, неизвестные категории или статусы и повреждённые
   CSV/JSON-записи. Добавьте проверки, специфичные для варианта, например
   формат email, телефона, координат или доменного идентификатора.
5. **С ИИ-агентом.** Предпочтительно поручить агенту подготовить
   детерминированный генератор. Передайте схему, объём, фиксированный `seed`,
   распределение случаев и требуемые форматы; код ETL не запрашивайте.
6. **Проверка результата.** Проверьте схему, объём, разнообразие записей,
   распределение случаев и ожидаемое число записей каждого типа. Генератор
   дважды запустите с одним `seed` и сопоставьте контрольные суммы.
7. **Самостоятельно.** Реализуйте Extract для CSV и JSON и параллельное чтение
   файлов. Ошибка одной записи не должна останавливать обработку остальных;
   недоступный или полностью повреждённый файл отражается в состоянии импорта.
8. **Самостоятельно.** Реализуйте Transform: очистку строк, раскрытие
   сокращений, преобразование чисел и дат, сверку дублирующих полей,
   нормализацию по справочникам и дедупликацию. Конкретные правила должны
   соответствовать выбранному варианту и таблице соответствия.
9. **Самостоятельно.** Реализуйте Load в PostgreSQL, UPSERT, запись метрик и
   DLQ с полями `sourceFile`, `rowNumber`, `safeIdentifier`, `stage`, `reason`.
10. **Проверка результата.** Запустите повторный импорт и подтвердите
    идемпотентность: число логических записей не увеличивается, а более новые
    версии обновляются по принятому правилу.

## Командная работа и Git

1. Создайте простую Kanban-доску в GitHub Projects или JetBrains YouTrack с
   колонками `To do`, `In progress`, `Done` и 6–10 задачами.
2. Для каждой задачи укажите исполнителя и короткие проверяемые критерии
   готовности.
3. Выполняйте крупные задачи в отдельных feature branches и связывайте задачи
   с коммитами или pull/merge requests.
4. Каждый участник должен провести не менее одного peer review изменения
   другого участника.
5. Добавьте в корень проекта `AGENTS.md`: вариант и назначение проекта,
   архитектурные ограничения, разрешённые области изменений, версии и
   зависимости, команды сборки/тестов и запрет на работу с секретами.

История Git должна позволять увидеть вклад каждого участника. Количество
коммитов само по себе не является показателем качества.

## Работа с ИИ-агентом

Помимо генератора грязных данных, ИИ-агент разрешён для декомпозиции задач,
тест-плана, тестов, анализа ошибок, code review и не более одной ограниченной
некритичной production-задачи на участника. Такой задачей может быть отдельный
DTO, один HTTP handler либо преобразование ответа в JSON по уже утверждённому
контракту.

Команда самостоятельно утверждает предметную модель, архитектуру, схему БД,
правила ETL и идемпотентности. Агенту нельзя поручать проект целиком или
одновременно генерировать Extract, Transform, Load и сервер. Для каждой
разрешённой задачи сохраните:

- точную постановку с целью, границами, ограничениями и критериями приёмки;
- существенный ответ или diff агента;
- выполненные команды и результаты тестов;
- принятые и отклонённые предложения;
- окончательные исправления команды.

Фиксированная персона вроде «ты — senior Java-разработчик» не требуется.
Важнее фактический контекст репозитория, интерфейсы, ограничения и проверяемый
результат.

## Дополнительная задача: собственный генератор и большой набор

За решение сверх обязательных требований можно получить до 5 дополнительных
баллов:

- до 3 баллов — параметризованный детерминированный генератор с фиксированным
  `seed`, настраиваемыми объёмом и распределением ошибок, выходом в CSV/JSON и
  автоматически формируемой ожидаемой сводкой;
- ещё до 2 баллов — воспроизводимый прогон на выбранном объёме от 10 000 до
  100 000 строк с замером времени обработки и сверкой метрик на входе, в БД и
  DLQ.

Использовать LLM или coding agent для подготовки генератора разрешено, но
команда обязана проверить его код и результаты. Итоговая оценка не превышает
100; бонус не компенсирует отсутствующий обязательный результат.

## Организация четырёх академических часов

| Этап | Время | Результат контрольной точки |
|---|---:|---|
| Вариант, контракт, архитектура и Kanban | 30 минут | утверждены mapping, API, схема БД, правила ETL и задачи |
| Вертикальный срез HTTP → JDBC | 60 минут | сервер запускается, health и одна операция БД работают |
| ETL и интеграция | 60 минут | выполняется импорт проверенного набора |
| Тестирование и демонстрация | 30 минут | пройден обязательный сценарий и подготовлены доказательства |

Контактные часы задают контрольные точки. Подготовка данных, завершение кода и
документации могут выполняться в рамках самостоятельной работы.

## Обязательный сценарий демонстрации

1. Назвать выбранный вариант и показать таблицу соответствия полей.
2. Собрать проект и запустить PostgreSQL и Java-сервер по инструкции.
3. Создать импорт минимум трёх файлов через `POST /imports` и получить
   `importId`.
4. Показать смену состояния и согласованные метрики через
   `GET /imports/{id}`.
5. Показать очищенные данные через `GET /records` с общим и доменным фильтрами,
   а отклонённые записи — в DLQ.
6. Повторить импорт и подтвердить отсутствие новых логических дубликатов.
7. Показать Kanban, историю Git, peer review и один пример исправления
   результата ИИ-агента.
