## Цель

Собрать изученные средства Java в одно серверное приложение: получить грязные данные из нескольких источников, выполнить конкурентный ETL, сохранить очищенный результат в PostgreSQL и предоставить доступ к импорту и данным через небольшой HTTP API. ИИ-агент используется для ограниченных задач, а команда проверяет каждый полученный артефакт.

Перед началом изучите инструкции по [генерации данных](../resources/llm-prompts/data-generation.md), [цепочкам промптов](../resources/llm-prompts/prompt-chains.md), [тестированию](../resources/llm-prompts/testing-and-debugging.md) и [работе coding agent с репозиторием](../resources/llm-prompts/README.md).

## Предметная модель

Каждая строка прайс-листа содержит:

| Поле | Требование |
|---|---|
| `sku` | непустой артикул товара |
| `name` | непустое название |
| `category` | исходная категория или значение для нормализации по справочнику |
| `price` | положительное десятичное число |
| `currency` | трёхбуквенный код валюты |
| `effectiveDate` | дата действия цены |
| `supplier` | непустой идентификатор или название поставщика |

Логический ключ предложения — пара `(supplier, sku)`. При нескольких корректных записях сохраняется запись с наиболее поздней `effectiveDate`. Конфликтующие записи с одинаковым ключом и датой должны обрабатываться по явно документированному детерминированному правилу.

## Обязательная архитектура

- Java 17 или новее, Maven либо Gradle;
- Jackson для JSON и PostgreSQL через JDBC;
- `ExecutorService` для параллельного чтения нескольких файлов;
- `JDK HttpServer` из модуля `jdk.httpserver`, без обязательного серверного фреймворка;
- конфигурация подключения к БД и каталога импорта вне исходного кода;
- локальный файл курсов валют и локальный справочник категорий для воспроизводимого запуска;
- логирование этапов и DLQ-файл для отклонённых записей.

Допускается переиспользовать и доработать код лабораторных №7–14. Простое копирование нескольких несвязанных проектов без единого запуска не считается интеграцией.

## База данных

Минимальная схема содержит:

- `suppliers` — поставщики с уникальным именем или кодом;
- `offers` — очищенные предложения с исходной валютой и ценой, ценой в RUB, датой и внешним ключом поставщика; комбинация `(supplier_id, sku)` уникальна;
- `import_runs` — идентификатор импорта, состояние, время начала и завершения и итоговые метрики.

Загрузка выполняется через `PreparedStatement` и UPSERT. Повторный импорт одинаковых файлов не создаёт новых логических предложений и не искажает метрики.

## HTTP API

Все ответы, кроме пустых технических ответов, имеют `Content-Type: application/json; charset=utf-8`.

### `POST /imports`

Тело запроса:

```json
{
  "files": ["supplier-a.csv", "supplier-b.json"]
}
```

Разрешены только имена файлов внутри настроенного каталога `data/inbox`; абсолютные пути и выход через `..` отклоняются. Сервер возвращает `202 Accepted`:

```json
{
  "importId": "generated-id",
  "status": "QUEUED"
}
```

### `GET /imports/{id}`

Возвращает одно из состояний `QUEUED`, `RUNNING`, `COMPLETED`, `COMPLETED_WITH_ERRORS`, `FAILED` и метрики `extracted`, `valid`, `rejected`, `duplicates`, `inserted`, `updated`.

### `GET /offers`

Возвращает очищенные предложения. Поддерживаются необязательные параметры `sku`, `supplier` и `limit`; некорректный `limit` приводит к ответу `400`, а не к необработанному исключению.

### `GET /health`

Возвращает состояние приложения и доступности PostgreSQL. В ответе не должно быть строки подключения, пароля или другой чувствительной конфигурации.

Аутентификация для учебного проекта не требуется. Неизвестный маршрут возвращает `404`, неверный запрос — `400`, внутренняя ошибка — `500` с безопасным сообщением и полной диагностикой только в журнале.

## ETL и тестовые данные

1. **Самостоятельно.** Зафиксируйте схему, правила валидности и распределение тестовых случаев. Набор должен содержать не менее трёх файлов обоих форматов, не менее 1000 записей суммарно, межфайловые дубликаты и минимум пять классов ошибок: пропуски, неверные цены, валюты, даты и повреждённые CSV/JSON-записи.
2. **С ИИ-агентом.** Передайте схему, объём, фиксированный `seed` и распределение случаев. Поручите создать детерминированный генератор входных файлов и ожидаемую сводку; код ETL не запрашивайте.
3. **Проверка результата.** Дважды запустите генератор с одним `seed`. Проверьте схему, объём, распределение, контрольные суммы и ожидаемое число каждого типа записей.
4. **Самостоятельно.** Реализуйте Extract для CSV и JSON и параллельное чтение файлов. Ошибка одной записи фиксируется, но не останавливает остальные корректные записи; недоступный или полностью повреждённый файл отражается в состоянии импорта.
5. **Самостоятельно.** Реализуйте Transform: очистку пробелов, нормализацию валюты и даты, проверку положительной цены, нормализацию категории по справочнику, пересчёт в RUB по локальным курсам и дедупликацию.
6. **Самостоятельно.** Реализуйте Load в PostgreSQL, UPSERT, запись метрик и DLQ с полями `sourceFile`, `rowNumber`, `safeIdentifier`, `stage`, `reason`.
7. **Проверка результата.** Запустите повторный импорт и подтвердите идемпотентность: число логических предложений не увеличивается, а более новые цены обновляются по правилу проекта.

## Командная работа и Git

1. Создайте простую Kanban-доску в GitHub Projects или JetBrains YouTrack с колонками `To do`, `In progress`, `Done` и 6–10 задачами.
2. Для каждой задачи укажите исполнителя и короткие проверяемые критерии готовности.
3. Выполняйте крупные задачи в отдельных feature branches и связывайте задачи с коммитами или pull/merge requests.
4. Каждый участник должен провести не менее одного peer review изменения другого участника.
5. Добавьте в корень проекта `AGENTS.md`: назначение проекта, архитектурные ограничения, разрешённые области изменений, версии и зависимости, команды сборки/тестов и запрет на работу с секретами.

История Git должна позволять увидеть вклад каждого участника. Количество коммитов само по себе не является показателем качества.

## Работа с ИИ-агентом

Помимо генератора грязных данных, ИИ-агент разрешён для декомпозиции задач, тест-плана, тестов, анализа ошибок, code review и не более одной ограниченной некритичной production-задачи на участника. Такой задачей может быть отдельный DTO, один HTTP handler либо преобразование ответа в JSON по уже утверждённому контракту.

Команда самостоятельно утверждает архитектуру, схему БД, правила ETL и идемпотентности. Агенту нельзя поручать проект целиком или одновременно генерировать Extract, Transform, Load и сервер. Для каждой разрешённой задачи сохраните:

- точную постановку с целью, границами, ограничениями и критериями приёмки;
- существенный ответ или diff агента;
- выполненные команды и результаты тестов;
- принятые и отклонённые предложения;
- окончательные исправления команды.

Фиксированная персона вроде «ты — senior Java-разработчик» не требуется. Важнее фактический контекст репозитория, интерфейсы, ограничения и проверяемый результат.

## Уровневые задания КРМ

Обязательная архитектура и сценарий демонстрации измеряют **Б**. Задания ниже преподаватель назначает до начала проекта как отдельную часть самостоятельной работы. Они не заменяют обязательный результат и оцениваются накопительно по [модели измерения уровней КРМ](../methodical-guidelines/teachers-assessment/krm-level-model.md).

### C-PL2.1 — Модель памяти и высокая конкурентность

1. Подготовьте воспроизводимый стресс-сценарий одновременного запуска не менее 100 импортов на наборе от 1 000 000 записей.
2. Обоснуйте видимость и атомарность изменяемого общего состояния через Java Memory Model; автоматическим тестом исключите потерянные обновления, гонки статусов и двойное завершение импорта.
3. Снимите JFR-профиль и GC logs для исходной конфигурации. Зафиксируйте throughput, latency, паузы GC и пиковую память минимум в трёх прогонах после прогрева.
4. Измените обоснованную настройку памяти/GC или структуру выделения объектов и повторите измерения. Оптимизация не должна нарушить согласованность результата.

Полное выполнение при подтверждённом Б фиксирует `PL-2.1 — С достигнут`.

### P-PL2.1 — Специализированное хранилище

1. Выберите специализированное хранилище, оправданное сценарием чтения проекта, например Elasticsearch для полнотекстового поиска предложений.
2. Реализуйте отдельный адаптер и согласованный механизм синхронизации с PostgreSQL; откажите безопасно при недоступности хранилища.
3. На воспроизводимой нагрузке сравните релевантную операцию со стартовым решением по корректности, latency и потреблению ресурсов.
4. Обоснуйте модель данных, индексы, границы согласованности и условия, при которых интеграция эффективна или избыточна.

Полное выполнение фиксирует `PL-2.1 — П достигнут` только после Б и С.

### C-PL2.2 — Фреймворки больших данных

1. Реализуйте эквивалентный пакетный этап в Apache Spark или Hadoop MapReduce и покажите адаптацию алгоритма с малого набора на набор не менее 1 000 000 записей.
2. Организуйте потоковое поступление через Kafka или обработку во Flink с детерминированной обработкой повторно доставленных событий.
3. Интегрируйте результат с ClickHouse или Greenplum и выполните минимум один воспроизводимый аналитический запрос.
4. Автоматически сопоставьте результат с обязательным ETL, соберите throughput/latency/error metrics и продемонстрируйте восстановление после контролируемого перезапуска.

Это единое задание заменяет пару C13-PL2.2 + C14-PL2.2. Полное выполнение при подтверждённом Б фиксирует `PL-2.2 — С достигнут`.

### P-PL2.2 — Профилирование и инфраструктура

1. На репрезентативной нагрузке профилируйте пакетный и потоковый ETL и локализуйте подтверждённое измерениями узкое место.
2. Выполните минимум одну алгоритмическую и одну инфраструктурную оптимизацию; сравните минимум три прогона до и после без изменения бизнес-результата.
3. Подготовьте воспроизводимое развёртывание, health checks, мониторинг, резервирование конфигурации и инструкцию восстановления после отказа компонента.
4. Самостоятельно проведите полный цикл `развёртывание → нагрузка → отказ → восстановление → повторная проверка` и приложите журналы и метрики.

Полное выполнение фиксирует `PL-2.2 — П достигнут` только после Б и С.

### P-LLM5.2 — Комплексный пайплайн

Используйте только модель и инструментальный интерфейс, которые действительно предоставляют attention и logits. Реализуйте комплексный пайплайн минимум с одной условной ветвью, сохранением выходов этапов и машинными проверками переходов. Для фиксированного набора случаев сопоставьте изменения attention, logits и итоговых выходов с успешными и ошибочными результатами, сформулируйте ограничения интерпретации и проверьте воспроизводимость анализа. Если доступный инструмент не возвращает attention или logits, преподаватель не назначает это задание и П по LLM-5.2 не фиксируется.

Полное выполнение фиксирует `LLM-5.2 — П достигнут` только после Б и С, подтверждённого заданием C15-LLM5.2.

## Дополнительная задача

Можно реализовать REST-клиент внешнего сервиса курсов валют. Он должен иметь timeout, проверку HTTP status и тела ответа, cache либо локальный fixture и автоматический fallback на локальные курсы. Внешний сервис не должен быть условием успешной демонстрации. За задачу начисляется до 5 дополнительных баллов, но итоговая оценка не превышает 100 и обязательные дефекты ею не компенсируются.

## Организация четырёх академических часов

| Этап | Время | Результат контрольной точки |
|---|---:|---|
| Контракт, архитектура и Kanban | 30 минут | утверждены API, схема БД, правила ETL и задачи |
| Вертикальный срез HTTP → JDBC | 60 минут | сервер запускается, health и одна операция БД работают |
| ETL и интеграция | 60 минут | выполняется импорт проверенного набора |
| Тестирование и демонстрация | 30 минут | пройден обязательный сценарий и подготовлены доказательства |

Контактные часы задают контрольные точки. Подготовка данных, завершение кода и документации могут выполняться в рамках самостоятельной работы.

## Обязательный сценарий демонстрации

1. Собрать проект и запустить PostgreSQL и Java-сервер по инструкции.
2. Создать импорт минимум трёх файлов через `POST /imports` и получить `importId`.
3. Показать смену состояния и согласованные метрики через `GET /imports/{id}`.
4. Показать очищенные данные через `GET /offers` и отклонённые записи в DLQ.
5. Повторить импорт и подтвердить отсутствие новых логических дубликатов.
6. Показать Kanban, историю Git, peer review и один пример исправления результата ИИ-агента.
