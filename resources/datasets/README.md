# Наборы данных для лабораторных работ (ООП, Java)

Комплект из **30 CSV-файлов** — по одному набору данных для каждого из 30 индивидуальных
проектов из документа «Экзаменационные вопросы и 30 индивидуальных проектов — ООП».
Каждый набор моделирует «сырые», неидеальные данные, с которыми студент должен научиться
работать при разработке классов чтения/валидации/обработки данных (парсинг, проверка,
приведение типов, обработка ошибок, коллекции, потоки, БД).

## Структура

- 30 файлов `NN_имя_проекта.csv` (например, `01_biblioteka.csv`, `12_avtoservis.csv` и т.д.) —
  соответствуют номерам проектов 1–30 из таблицы вариантов.
- Каждый файл содержит **200 строк данных** + строка заголовка.
- Кодировка: UTF-8 с BOM (`utf-8-sig`) — корректно открывается в Excel и всех современных
  средах разработки.
- Разделитель: запятая (`,`); поля, содержащие запятую (например, адрес), заключены в
  кавычки по стандарту CSV (RFC 4180) — это намеренно, чтобы студенты тренировались в
  корректном разборе CSV, а не просто разбивали строку по запятой.
- Файл `_summary.txt` — сводная статистика по каждому набору (доля сокращений, ошибок,
  пропусков). Файл не должен попасть к студентам.

## Структура столбцов (18 колонок, избыточный набор полей)

Все наборы имеют одинаковую по смыслу структуру из 18 столбцов, где часть полей —
избыточные/дублирующие для тренировки нормализации и очистки данных:

| № | Столбец | Назначение |
|---|---------|------------|
| 1 | `ID` | Порядковый номер записи (техническое поле, никогда не искажается) |
| 2 | `Код записи` | Бизнес-ключ вида `ПРЕФИКС-000001` (никогда не искажается) |
| 3 | Основное наименование (зависит от домена) | Главная сущность записи |
| 4 | Основное наименование (повтор) | **Избыточное** дублирующее поле — должно совпадать с (3), но может расходиться из-за внесённых искажений |
| 5 | Роль-персона (зависит от домена) | Например «Автор», «Клиент», «Врач» |
| 6 | Категория (зависит от домена) | Категориальное поле — **допускает корректные сокращения** |
| 7 | Подкатегория (зависит от домена) | **Избыточное** классифицирующее поле, обычный список значений |
| 8 | Город | Общий для всех доменов список городов РФ — **допускает корректные сокращения** |
| 9 | Адрес | **Избыточно** пересекается с городом; содержит встроенные запятые (тест на корректный CSV-парсинг) |
| 10 | Телефон | Формат `+7-9XX-XXX-XX-XX` |
| 11 | Email | |
| 12 | Статус (зависит от домена) | Категориальное поле — **допускает корректные сокращения** |
| 13 | Количественный показатель (зависит от домена) | Числовое поле |
| 14 | Стоимостной показатель (зависит от домена) | Числовое поле |
| 15 | Дата создания записи | Формат `ДД.ММ.ГГГГ` |
| 16 | Дата последнего обновления | **Избыточное** поле, формат `ДД.ММ.ГГГГ` |
| 17 | Система-источник | **Избыточное** метаданное (например, «1С:Предприятие», «CRM Bitrix24») |
| 18 | Примечания | Свободный текст, часто отсутствует |

Конкретные подписи столбцов (например, «Название издания», «Жанр», «Статус экземпляра»)
адаптированы под предметную область каждого проекта — см. первую строку каждого файла.

## Методология внесения «шума» в данные

Для каждого набора из 200 строк × 16 «редактируемых» полей (все столбцы, кроме `ID` и
`Код записи`) случайным образом внесены три вида изменений без пересечения ячеек:

1. **Корректные сокращения (~25%)** — применяются только к столбцам `Категория`,
   `Статус` и `Город`. Полное значение заменяется его официальным сокращением
   (например, «Художественная литература» → «Худ. лит-ра», «Москва» → «Мск»,
   «В обработке» → «В обр.»). Это **не ошибка** — сокращение корректно и однозначно
   соответствует полному значению; студенты должны научиться распознавать и
   разворачивать такие сокращения через таблицу соответствий.
2. **Некорректно заполненные данные (≈3–4% от всех редактируемых ячеек)** — значение
   заменено на заведомо неверное для данного типа поля:
   - текстовые поля → мусорные токены («N/A», «???», «ОШИБКА_ДАННЫХ», «12345XYZ» и т.п.);
   - категориальные поля → значение не из справочника («Неизвестная категория», «###»);
   - числовые поля → нечисловая строка («много», «н/д») или явно некорректное значение;
   - даты → несуществующая дата («31.02.2026», «00.00.0000», «2026-13-45»);
   - телефон → неверный формат (буквы вместо цифр, неверное количество разрядов);
   - email → неверный формат (без «@», двойные точки).
3. **Неполно заполненные данные (≈4–5% от всех редактируемых ячеек)** — значение либо
   полностью пустое, либо обрезано (например, только первое слово текста, только год
   у даты, email без домена, телефон без последних цифр).

Фактические доли по каждому набору приведены в `_summary.txt` (они находятся в пределах
3.0–4.0% для ошибок и 4.1–5.0% для пропусков, как и требуется в задании).

## Как это использовать в лабораторных работах

Эти датасеты подходят для отработки следующих тем курса:
- чтение и парсинг CSV (в т.ч. корректная обработка кавычек и встроенных запятых);
- валидация и приведение типов (число, дата, телефон, email);
- обработка исключений при некорректных данных;
- нормализация сокращений через справочники (`Map<String, String>`);
- обработка пропущенных значений (значения по умолчанию, пропуск записи, логирование);
- устранение избыточности (сверка полей-дублей, выбор канонического значения);
- дальнейшая загрузка очищенных данных в коллекции/БД для последующих лабораторных
  (по мере развития индивидуального проекта).

## Воспроизводимость

Все наборы сгенерированы одним Python-скриптом с фиксированным `random.seed(42)`,
что обеспечивает воспроизводимость при необходимости регенерации или расширения набора.
