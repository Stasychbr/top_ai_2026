# Лабораторная 16. Мини-проект: проектирование и начало реализации

## Тема

**Командный мини-проект «Сервис загрузки и очистки прайс-листов».**

## Задачи

Задание (2–3 человека в команде)

1. Постановка задачи: разработать консольное Java-приложение, которое:
- Принимает на вход CSV или JSON-файл с прайс-листом поставщика (поля: артикул, название, категория, цена, валюта, дата, поставщик).
- Выполняет ETL: очистку (удаление дубликатов), нормализацию (валюта к единому стандарту — RUB по курсу ЦБ на дату), обогащение (добавление категории из справочника).
- Загружает результат в БД.
- Поддерживает конкурентную загрузку нескольких файлов (ExecutorService).
2. На этом занятии: распределение ролей, проектирование архитектуры (диаграмма классов), соглашение об интерфейсах между модулями. Инициализация Git-репозитория, настройка Maven/Gradle, подключение зависимостей.
3. Составить промпт для генерации каркаса проекта — структуры пакетов и интерфейсов. Обсудить результат, скорректировать.


## Стек технологий

- **Язык:** Java (версия 17+)
- **IDE:** JetBrains IntelliJ IDEA Community Edition
- **VCS:** Git
- **ИИ-помощник:** GitHub Copilot (или аналоги)

## Инструкция по выполнению

### 1. Постановка задачи (бизнес-контекст)

Разработать консольное Java-приложение для агрегации данных от поставщиков маркетплейса.
Приложение должно работать в режиме пайплайна:

- Источник: CSV или JSON-файл с прайс-листом поставщика.
- Поля входных данных: sku (артикул), title (название), rawCategory, price, currency, date, vendor.
- ETL-процесс:
  - Extract: Чтение файла.
  - Transform: Очистка, нормализация валюты к RUB по курсу ЦБ на дату из строки товара, обогащение категории через справочник соответствия (rawCategory -> normalizedCategoryId).
  - Load: Сохранение очищенных записей в реляционную БД (PostgreSQL/H2).
- Масштабируемость: Поддержка конкурентной обработки нескольких файлов одновременно через ExecutorService.

### 2. Задачи текущего занятия 

Цель этапа — подготовить фундамент разработки («нулевой спринт»), чтобы избежать хаоса при написании кода.

Шаг 1. Распределение ролей (RACI-матрица)
Команда из 2–3 человек распределяет зоны ответственности:

| Роль                   | Участник(и) | Ответственность                                                                                                            |
| :--------------------- | :---------- | :------------------------------------------------------------------------------------------------------------------------- |
| Архитектор / Tech Lead | 1 чел.      | Утверждение диаграммы классов, контроль соблюдения контрактов интерфейсов, ревью MR/PR.                                    |
| Разработчик ядра ETL   | 1–2 чел.    | Реализация парсеров (CSV/JSON), сервисов очистки, логики нормализации валют.                                               |
| DevOps / Интегратор    | 1 чел.      | Настройка CI/CD (GitHub Actions/GitLab CI), оркестрация Docker Compose (App + DB), управление миграциями Liquibase/Flyway. |

Шаг 2. Проектирование архитектуры (Диаграмма классов)
Необходимо спроектировать систему согласно принципам чистой архитектуры (Clean Architecture). Основные пакеты:

1. com.marketplace.etl.config: Конфигурация Spring Boot / DI-контейнера, настройки подключения к БД и API курсов валют.
2. com.marketplace.etl.model.dto: DTO-классы, повторяющие структуру сырых входящих файлов (RawPriceItem).
3. com.marketplace.etl.model.entity: JPA-сущности для базы данных (NormalizedPriceItem).
4. com.marketplace.etl.parser:
Интерфейс PriceListParser { List<RawPriceItem> parse(Path file); }
Реализации: CsvPriceListParser, JsonPriceListParser.
5. com.marketplace.etl.service:
DuplicationRemovalService: удаление дублей по комбинации (sku, vendor).
6. CurrencyNormalizationService: запрос курса ЦБ через внешний API (или заглушку) и конвертация.
CategoryEnrichmentService: маппинг категорий.
com.marketplace.etl.repository: Spring Data JPA репозиторий.
7. com.marketplace.etl.runner: Точка входа приложения (CommandLineRunner), координатор потоков ExecutorService.

Шаг 3. Соглашение об интерфейсах (Contract-first)
До написания бизнес-логики команда фиксирует сигнатуры ключевых методов.

Шаг 4. Инициализация инфраструктуры
- VCS: Создание приватного репозитория. Настройка .gitignore (включить /out/, *.iml, /target/, .idea/, файлы переменных окружения .env).
- Сборка: Инициализация Maven или Gradle.
- База данных: Подключение PostgreSQL  в зависимостях.
- API курсов: Добавление зависимости HTTP-клиента.

### 3. Генерация через ИИ (Copilot)

## Контрольные вопросы

1.  Каким образом разделение одной сложной задачи («сделать REST-клиент») на три последовательных промпта (архитектура → модель → реализация) влияет на качество итогового кода по сравнению с попыткой получить всё решение в одном сообщении?
2. В чем практическая разница между запросом «напиши класс Order» и «ты senior Java-разработчик, напиши класс Order, соблюдая стандарты нашей компании»? Какие скрытые параметры (помимо синтаксиса) задает указание роли?
3. Почему при генерации HttpClient ИИ часто забывает про таймауты (timeout()) или блокировку потоков (join()/Future), если это явно не прописано в ограничениях промпта, несмотря на то что это стандарт промышленной разработки?
4. Оцените эффективность использования условий вида «используй record, если JDK 17+, иначе обычный класс». Способна ли модель корректно выполнить эту проверку самостоятельно, или она склонна генерировать либо только один вариант, либо дублирующий код?
5. Какой из пунктов чек-листа (компиляция, обработка ошибок, логирование, безопасность, нейминг) является наиболее критичным для продакшен-кода и почему ИИ-модели чаще всего «проваливают» именно пункт безопасности (например, утечка токенов или SQL-инъекции), даже если они знают правила написания безопасного кода теоретически?

## Критерии оценки

Полнота и структурированность файла README.md, в котором должны быть дословно зафиксированы все промпты (включая указание роли эксперта), архитектурные решения и сгенерированные фрагменты кода; глубина проведенного code review по чек-листу (проверка на компилируемость, обработку исключений, логирование, безопасность и нейминг) с обоснованными выводами о качестве каждого фрагмента; корректность реализации цепочки из трех промптов, где каждый последующий шаг явно опирается на результат предыдущего без потери контекста; качество сравнительного анализа двух ИИ-моделей в табличном виде с выделением объективных различий в синтаксисе и стиле написания кода; а также чистота коммита в Git, содержащего только аналитическую документацию без исходного кода проекта.

## Варианты

Лабораторная 16. Мини-проект: проектирование и начало реализации (20 вариантов)

Общая концепция: Командный мини-проект (2–3 человека). Разработка консольного Java-приложения для ETL-обработки данных в конкретной предметной области. На текущем этапе — распределение ролей, проектирование архитектуры (диаграмма классов), соглашение об интерфейсах, инициализация Git/Maven/Gradle.

1. Банковское дело
Продукт: Агрегатор выписок по счетам из разных банков (CSV).
ETL: Очистка транзакций от тестовых платежей банка, нормализация валют к USD, обогащение MCC-кодов категориями трат («Еда», «Транспорт»).
БД: Транзакционный реестр transactions.
Интерфейс: ExchangeRateProvider.getRate(String currency, LocalDate date).
2. Автомобильный транспорт
Продукт: Парсер объявлений о продаже авто с досок (JSON/XML).
ETL: Удаление дублей VIN-номеров, нормализация пробега (мили -> км), обогащение комплектации через справочник опций.
БД: Каталог автомобилей vehicle_catalog.
Интерфейс: MileageNormalizer.normalize(double value, String unit).
3. Электронная коммерция
Продукт: Сбор товарных фидов от поставщиков маркетплейса (YML/CSV).
ETL: Дедупликация товаров по артикулу поставщика, конвертация цен со скидками, привязка к складам (FBS/FBO).
БД: Пул остатков stock_pool.
Интерфейс: DiscountCalculator.calculateFinalPrice(BigDecimal basePrice, BigDecimal discountPercent).
4. Социальные сети
Продукт: Миграция профилей пользователей из старого CSV-дампа в новую систему.
ETL: Хэширование паролей (SHA-256/Bcrypt), валидация email, дедупликация по почте, генерация дефолтных аватаров.
БД: Таблица users профиля.
Интерфейс: PasswordHasher.hash(String plainText).
5. Геолокация и картография
Продукт: Загрузка POI (точек интереса) из GeoJSON файлов партнеров (кафе, заправки).
ETL: Фильтрация точек вне зоны покрытия, расчет bounding box, перевод координат WGS84 в локальную проекцию.
БД: Spatial БД PostGIS (poi_points).
Интерфейс: CoordinateTransformer.transform(WgsPoint point).
6. Университет и образование
Продукт: Импорт списков студентов и оценок из ведомостей Excel/CSV.
ETL: Приведение ФИО к единому регистру, проверка формата номера зачетки, расчет среднего балла (GPA).
БД: Реестр успеваемости student_grades.
Интерфейс: GradeValidator.validate(int grade) (проверка диапазона 2–5).
7. Видеоигры
Продукт: Агрегация статистики игроков из логов игровых серверов (JSON Lines).
ETL: Обогащение ID игрока никнеймом через API Steam/Epic, фильтрация читерских показателей (K/D > 100), группировка по сезонам.
БД: Лидерборды leaderboards.
Интерфейс: CheatDetector.isSuspicious(GameStats stats).
8. Медицина
Продукт: Интеграция лабораторных анализов из систем клиник (HL7 или кастомный JSON).
ETL: Анонимизация ФИО пациентов, кодирование диагнозов по справочнику МКБ-10, приведение единиц измерения (ммоль/л -> мг/дл).
БД: Журнал анализов lab_results.
Интерфейс: DiagnosisCoder.encode(String textDescription).
9. Библиотечное дело
Продукт: Синхронизация каталога книг с внешними библиотечными системами (формат MarcXchange/XML).
ETL: Нормализация ISBN (удаление тире), объединение изданий одной книги под одним WorkID, извлечение года издания.
БД: Каталог фонда library_fund.
Интерфейс: IsbnCleaner.canonicalize(String rawIsbn).
10. Космонавтика

Интерфейс: DeviceClassifier.predictType(PowerCurve curve).
12. Ресторанный бизнес
