# Лабораторная 17. Мини-проект: реализация, тестирование и защита

## Тема

**Завершение мини-проекта. Демонстрация и защита.**

## Задачи

Задание (2–3 человека в команде)

1. Завершить реализацию всех модулей согласно архитектуре.
2. Интегрировать модули, провести сквозное тестирование на тестовых данных (не менее 3 файлов разных форматов с разными валютами и заведомо некорректными записями).
3. Подготовить README с описанием архитектуры, инструкцией по запуску и документированными промптами ИИ.
4. Защита: каждая команда проводит живую демонстрацию — запуск приложения на тестовых данных, демонстрация результатов в БД, ответы на вопросы. Оценивается: корректность ETL, стабильность при конкурентной загрузке, качество кода (code review преподавателем), полнота документации.
5. Каждый студент описывает, на каких этапах ИИ помог, на каких — помешал, и чему он научился в части взаимодействия с ИИ-инструментами.

## Стек технологий

- **Язык:** Java (версия 17+)
- **IDE:** JetBrains IntelliJ IDEA Community Edition
- **VCS:** Git
- **ИИ-помощник:** GitHub Copilot (или аналоги)

## Инструкция по выполнению

### 1. Завершение реализации модулей

Команда реализует бизнес-логику согласно архитектуре, спроектированной на предыдущем занятии:

- Парсеры (CsvPriceListParser, JsonPriceListParser): обработка вложенных структур, экранирования кавычек в CSV.
- Сервисы очистки (DuplicationRemovalService): реализация через HashSet с переопределенным equals()/hashCode() или использование Java Stream API distinct().
- Нормализация данных:
  - Валюты: интеграция с заглушкой API курсов ЦБ (возвращает фиксированные значения для USD/EUR/CNY) или реальным публичным API.
  - Категории: загрузка справочника из JSON-ресурса приложения при старте контекста.
- Конкурентность (ExecutorService): настройка пула потоков (размер = количество доступных ядер + 2), передача файлов в очередь задач, сбор результатов через CompletableFuture.
- Persistence: написание JPA-сущностей, аннотаций маппинга, миграций Liquibase/Flyway.

### 2. Интеграция и сквозное тестирование
Необходимо подготовить набор тестовых данных, имитирующих реальный хаос от поставщиков.

Матрица тестовых файлов (минимум 3 шт.):

| Имя файла                            | Формат | Особенности                                                                                                     |
| :----------------------------------- | :----- | :-------------------------------------------------------------------------------------------------------------- |
| supplier_a_usd.csv                   | CSV    | Корректные данные, валюта USD, дубликаты артикулов внутри файла.                                                |
| supplier_b_eur.json                  | JSON   | Данные в EUR, одна запись имеет некорректный формат даты, цена указана строкой "100,50" (запятая вместо точки). |
| supplier_c_mixed.xlsx_converted.json | JSON   | Смешанные валюты (RUB, GBP), пустые поля category, отрицательная цена (ошибка ввода).                           |
|                                      |        |                                                                                                                 |

Критерии успешного теста:

- Приложение не падает при встрече исключения в одной строке (использование паттерна Try-Catch внутри потока обработки записи).
-  Некорректные строки записываются в отдельный файл логов rejected_records.log с указанием причины ошибки и номера строки.
- В БД попадают только валидные, дедуплицированные записи с ценой, пересчитанной в RUB по курсу на дату поставки.
- При запуске конкурентной загрузки трех файлов одновременно итоговый баланс валютных конвертаций сходится, нет состояния гонки (Race Condition) при обновлении курса.

### 3. Подготовка документации (README.md)

Файл должен быть развернутым и служить единственным источником истины (Single Source of Truth).

Структура README:

- Описание проекта: Цель сервиса, схема ETL-конвейера (визуальная ASCII-диаграмма или скриншот диаграммы классов).
- Архитектура: Описание слоев (Controller -> Service -> Repository), используемых паттернов проектирования.
- Инструкция по запуску (Quick Start):
  -Требования (JDK 17+, Docker для БД).
  - Команды сборки (mvn clean package) и запуска (java -jar target/app.jar --input=./data).
  -Скрипты для поднятия локальной БД (docker-compose up -d).
- Тестовые сценарии: Как запустить тесты (mvn test), описание структуры папки /src/test/resources/fixtures.
- Раздел AI Transparency (Прозрачность ИИ):
  - Список использованных промптов.
  - Ссылки на конкретные коммиты, где код был сгенерирован ИИ.
  - Оценка качества генерации (процент кода, оставленного без изменений vs отрефакторенного).

### 4. Защита проекта (Демонстрация)
Формат защиты — живая сессия (15–20 минут на команду).

План демонстрации:

- Запуск окружения: Студенты показывают терминал со стартом БД и деплоем приложения.
- ETL в действии: Запуск приложения с передачей папки тестовых данных.
- Команда выводит содержимое консоли (логи INFO/WARN/ERROR).
- Показывают сформированный файл rejected_records.log.
- Проверка результата: Открытие клиента БД (DBeaver/DataGrip) и выполнение SQL-запросов:
- Нагрузочное тестирование (Stability check): Запуск скрипта, создающего 10+ временных файлов, демонстрация того, что ExecutorService корректно распределяет нагрузку CPU и память не течет (мониторинг через VisualVM/JConsole).

Оценка преподавателя :

- Корректность ETL (0–30 баллов): Верная логика очистки, отсутствие потерь данных, точность математических расчетов валют.
- Стабильность (0–20 баллов): Грамотная работа исключений, потокобезопасность, отсутствие зависаний при конкурентности.
- Качество кода (0–25 баллов): Читаемость, нейминг, покрытие юнит-тестами критической логики (валидаторы, конвертеры), соблюдение Code Style.
- Документация и DevOps (0–15 баллов): Полнота README, наличие .gitignore, конфигов CI/CD (опционально).
- Защита и ответы на вопросы (0–10 баллов): Глубина понимания собственного кода, умение объяснить архитектурные решения.

## Ожидаемый результат

По завершении работы каждая команда должна предоставить целостный, готовый к запуску программный продукт и пакет сопроводительной документации.

1. Программное обеспечение (Deliverable)
Ссылка на Git-репозиторий (GitHub / GitLab / Bitbucket), содержащий полный исходный код проекта.
Структура проекта: Соответствие архитектуре Spring Boot или чистой Java SE с использованием Maven/Gradle. Пакеты логически разделены (parser, service, repository, config).
Исполняемый артефакт: Корректно собранный .jar файл (fat-jar/executable jar), который запускается командой java -jar.
Конфигурация:
Настроенный пул потоков в ExecutorService.
Рабочее подключение к СУБД через JPA/JDBC.
Файл конфигурации (application.properties или application.yml) с вынесенными параметрами (пути к файлам, URL API курсов валют, настройки БД).
2. Данные и тестовый стенд
Папка /data: Набор из не менее чем 3-х тестовых файлов разных форматов (например, CSV, JSON, XML) с заведомо некорректными записями:
Нарушение типов данных (текст вместо числа).
Некорректные даты.
Отрицательные цены.
Пропущенные обязательные поля.
Дубликаты записей.
Состояние базы данных: После запуска приложения в целевой таблице должны находиться только валидные данные:
Все валюты приведены к RUB согласно курсу на дату операции.
Полностью отсутствуют дубликаты.
Категории нормализованы (вместо сырых строк — ID справочника).
В служебных таблицах/системах зафиксированы отбракованные записи с указанием причины ошибки.
3. Документация (Файл README.md)

Подробный файл в корне репозитория, включающий следующие разделы:

Описание архитектуры: Текстовое пояснение и диаграмма компонентов (C4 Model или упрощенная схема ETL-конвейера).
Инструкция по развертыванию (Quick Start):
Требования к окружению (версия JDK).
Команда сборки (mvn clean install).
Команда запуска с примерами аргументов командной строки.
Инструкция по поднятию локальной БД (скрипты Docker Compose или миграции Liquibase).
Примеры входных и выходных данных: Образец сырой строки из файла и соответствующий ей SQL-запрос к базе после обработки.
Раздел «AI Transparency»: Документированные промпты, использованные для генерации кода, анализ их эффективности и список мест, где ИИ-код был переписан вручную.
4. Результаты защиты (Live Demo & Review)

В ходе демонстрации преподавателю демонстрируются:

Живой запуск: Запуск приложения на терминале преподавателя с передачей папки с данными.
Логи выполнения: Демонстрация консоли, где видно конкурентное выполнение задач (Thread names), обработка ошибок без падения всего пайплайна и итоговая статистика (processed/failed records).
Проверка БД: Прямое подключение к базе данных и выполнение проверочных SQL-запросов для подтверждения корректности расчетов.
Качество кода: Преподаватель открывает случайные файлы классов в IDE для оценки читаемости, нейминга и отсутствия закомментированного кода.
Ретроспектива команды: Устные ответы студентов на вопросы о том, как они распределяли задачи и какие сложности вызвал ИИ-инструмент при совместной работе.

Критерии зачета :
Проект считается выполненным успешно, если приложение завершает работу со статусом exit code 0, база данных содержит ожидаемые очищенные данные, документация позволяет стороннему разработчику запустить проект за 5 минут, а студенты способны объяснить любую строчку кода, включая сгенерированную ИИ.
