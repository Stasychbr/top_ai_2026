# Лабораторная 15. Итоговая лабораторная по промпт-инжинирингу

## Тема

**Составление и структурирование промптов. Цепочки промптов. Оценка качества сгенерированного кода.**

## Задачи

Задание (выполняется без написания кода — только промпты и анализ):

1. Составить промпт для генерации класса Order (поля: id, customerName, items, totalAmount, status, createdAt). Требования к промпту: указать роль (ты — senior Java-разработчик), формат вывода (полный код класса с аннотациями для Jackson), ограничения (использовать record, если Java 17+, либо обычный класс).
2. Составить цепочку из 3 промптов для задачи «спроектировать REST-клиент для внешнего API»:
   - Промпт 1: описать архитектуру классов (клиент, модель ответа, обработчик ошибок).
   - Промпт 2: на основе архитектуры сгенерировать код модели ответа.
   - Промпт 3: на основе архитектуры и модели сгенерировать код клиента с использованием HttpClient.
3. Для каждого сгенерированного ИИ фрагмента кода провести code review по чек-листу: компилируется ли, обработаны ли ошибки, есть ли логирование, соблюдены ли соглашения об именовании, нет ли уязвимостей.
4. Документировать все промпты и результаты анализа в README репозитория.
5. Сравнить ответы разных инструментов на один и тот же промпт.

## Стек технологий

- **Язык:** Java (версия 17+)
- **IDE:** JetBrains IntelliJ IDEA Community Edition
- **VCS:** Git
- **ИИ-помощник:** GitHub Copilot (или аналоги)

## Инструкция по выполнению

### 1. Подготовка

Cоздайте новую ветку:

```bash
git checkout -b feature/lab15-prompting
```

### 2. Оформления ответов

Используйте Markdown для оформления ответов ИИ. Блоки кода должны быть обернуты в ```.

### 3. Генерация через ИИ (Copilot)

При составлении промптов используйте технику ролевого инжиниринга («You are an expert in...» + «I want you to act as...» + «Output only code without explanations»).
Для цепочек сохраняйте контекст диалога либо копируйте предыдущие ответы целиком в новый запрос.

### 4. Работа с системой контроля версий

- Зафиксируйте изменения:

```bash
git status
git add README.md
git commit -m "docs(lab15): final prompt engineering analysis and code review report"
```

## Контрольные вопросы

1.  Каким образом разделение одной сложной задачи («сделать REST-клиент») на три последовательных промпта (архитектура → модель → реализация) влияет на качество итогового кода по сравнению с попыткой получить всё решение в одном сообщении?
2. В чем практическая разница между запросом «напиши класс Order» и «ты senior Java-разработчик, напиши класс Order, соблюдая стандарты нашей компании»? Какие скрытые параметры (помимо синтаксиса) задает указание роли?
3. Почему при генерации HttpClient ИИ часто забывает про таймауты (timeout()) или блокировку потоков (join()/Future), если это явно не прописано в ограничениях промпта, несмотря на то что это стандарт промышленной разработки?
4. Оцените эффективность использования условий вида «используй record, если JDK 17+, иначе обычный класс». Способна ли модель корректно выполнить эту проверку самостоятельно, или она склонна генерировать либо только один вариант, либо дублирующий код?
5. Какой из пунктов чек-листа (компиляция, обработка ошибок, логирование, безопасность, нейминг) является наиболее критичным для продакшен-кода и почему ИИ-модели чаще всего «проваливают» именно пункт безопасности (например, утечка токенов или SQL-инъекции), даже если они знают правила написания безопасного кода теоретически?

## Критерии оценки

Полнота и структурированность файла README.md, в котором должны быть дословно зафиксированы все промпты (включая указание роли эксперта), архитектурные решения и сгенерированные фрагменты кода; глубина проведенного code review по чек-листу (проверка на компилируемость, обработку исключений, логирование, безопасность и нейминг) с обоснованными выводами о качестве каждого фрагмента; корректность реализации цепочки из трех промптов, где каждый последующий шаг явно опирается на результат предыдущего без потери контекста; качество сравнительного анализа двух ИИ-моделей в табличном виде с выделением объективных различий в синтаксисе и стиле написания кода; а также чистота коммита в Git, содержащего только аналитическую документацию без исходного кода проекта.
