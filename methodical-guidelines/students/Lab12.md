# Лабораторная 12. JDBC: транзакции и пакетная обработка

## Тема

**Транзакции. Управление конкурентным доступом. Пакетные операции.**

## Задачи

1. Реализовать метод transferBooks(long fromAuthorId, long toAuthorId) — все книги одного автора переназначить другому. Вся операция — в одной транзакции.
2. Если у целевого автора уже есть книга с таким же ISBN — откатывать транзакцию целиком.
3. Реализовать метод batchInsert(List<Book> books) — вставка 10 000 записей в одной транзакции с использованием PreparedStatement.addBatch() и executeBatch().
4. Замерить время пакетной вставки vs построчной вставки.
5. Реализовать saveOrUpdate — если книга с таким ISBN уже существует, обновить её; иначе вставить (UPSERT на уровне Java-логики, затем через SQL ON CONFLICT).
6. Настроить HikariCP как пул соединений: сравнить время 100 последовательных запросов с пулом и без пула.
7. Сгенерировать SQL-запрос INSERT ... ON CONFLICT DO UPDATE для Book по текстовому описанию. Проверить синтаксис на реальной БД.

### 2. Транзакция переназначения (transferBooks)

- Реализовать метод void transferBooks(long fromAuthorId, long toAuthorId).
- Отключить автокоммит: conn.setAutoCommit(false).
- Обновить поле author_id у всех книг первого автора на ID второго.
- Если в процессе обновления возникнет ошибка (например, нарушение уникальности ISBN, если бизнес-логика подразумевает проверку), вызвать conn.rollback().
- При успехе — conn.commit(). Обязательно вернуть autoCommit в true в блоке finally.

### 3. Пакетная вставка (batchInsert)

- Реализовать метод для вставки 10 000 записей
- Использовать один PreparedStatement вне цикла
- Внутри цикла: stmt.set...(); stmt.addBatch();
- После цикла: int[] results = stmt.executeBatch();
- Весь батч должен быть обернут в одну транзакцию

### 4.Бенчмарк производительности

- Написать тест, сравнивающий время выполнения:

  - Вариант А: Вставка 10 000 строк по одной (каждая в своей транзакции).
  - Вариант Б: Пакетная вставка (п.3).

### 5. UPSERT (SaveOrUpdate)

- Реализовать логику «Сохранить или обновить».
- Сначала попробовать сделать это через SQL конструкцию INSERT ... ON CONFLICT(isbn) DO UPDATE SET title = EXCLUDED.title....
- Затем реализовать ту же логику вручную на Java: сначала SELECT, затем либо UPDATE, либо INSERT внутри транзакции.

### 6. Пул соединений (HikariCP)

- Добавить зависимость HikariCP.
- Сравнить время выполнения 100 простых запросов SELECT 1 при получении соединения каждый раз через DriverManager vs получение из пула HikariDataSource.

## Ожидаемый результат

Ссылка на репозиторий. Программа демонстрирует:

- Перенос книг работает атомарно (либо все переехали, либо ни одна).
- Пакетная вставка выполняется значительно быстрее построчной (в десятки раз).
- Время работы с пулом HikariCP меньше, чем без него (отсутствие накладных расходов на TCP handshake).
- Метод UPSERT не создает дублей при повторном запуске.

## Критерии оценки

Корректная настройка пула соединений HikariCP и сравнение его производительности с DriverManager, реализация метода переназначения книг строго внутри одной транзакции с использованием setAutoCommit(false), commit() и rollback() при возникновении SQLException или нарушении бизнес-правил (дубликаты ISBN), наличие эффективной пакетной вставки 10 000 записей через PreparedStatement.addBatch() и executeBatch() в рамках единой транзакции, предоставленный бенчмарк, наглядно демонстрирующий разницу во времени между пакетным и последовательным режимами, а также реализованный UPSERT-запрос (через SQL ON CONFLICT или Java-логику) и работа с ИИ для генерации синтаксиса MERGE/UPSERT.
