# Методические указания обучающимся

## Предисловие к упражнениям

Данные упражнения представляют собой практический цикл лабораторных работ по основам объектно-ориентированного программирования на языке Java.

Переходя от простого к сложному, мы закрепляем фундаментальные принципы разработки: инкапсуляцию данных через модификаторы доступа, гибкость проектирования с помощью перегрузки методов и конструкторов, а также управление общим состоянием класса через статические поля и фабричные методы. Каждое задание моделирует реальную предметную область — от банковского счета до управления космическими миссиями.

Особенностью данного цикла является интеграция инструментов искусственного интеллекта (GitHub Copilot) в процесс написания кода. ИИ-помощник выступает не как замена разработчика, а как инструмент для сравнения подходов:

Сначала студент реализует логику вручную, опираясь на требования ООП.
Затем аналогичная структура генерируется нейросетью по текстовому описанию или комментарию на естественном языке.
Итогом становится анализ полученного кода: оценка его читаемости, полноты созданных методов, соответствия стандартам оформления (Code Style) и выявление потенциальных ошибок, которые мог допустить ИИ.

Такой подход позволяет научиться грамотно ставить задачи искусственному интеллекту, критически оценивать результат генерации и понимать архитектурные различия между ручным написанием и машинным шаблоном, используя при этом современные инструменты разработки (IntelliJ IDEA) и систему контроля версий Git.

Каждое занятие имеет сквозную нумерацию, тему, краткое описание задачи и указание применяемого ИИ-инструмента. Лабораторные 1–15 выполняются индивидуально; 16–17 — итоговый мини-проект

## Предоставление отчетности

В целом по блоку упражнений ожидается представление результатов в виде единого сквозного портфолио, которое демонстрирует эволюцию навыков от простого написания кода к инженерному подходу.

Результат представляется в двух форматах: электронном (репозиторий и отчетность) и устном/визуальном (защита).

### 1. Электронное представление (Репозиторий-дневник)

Вместо набора разрозненных папок с лабами предоставляется один главный Git-репозиторий (или организация на GitHub), где каждая лабораторная работа — это логический этап развития проекта.

Структура репозитория:
Четкое ветвление (main, dev + feature branches для каждой темы: feature/lab2-overloading, feature/lab3-inheritance).
Семантическое версионирование тегов (v0.1.0 — базовый класс Book, v0.2.0 — добавлена статика и фабрика).
Наличие файла .gitignore и настроенной системы сборки (Maven/Gradle) во всем корне.
Эволюция кода:
Преподаватель должен видеть, как классы из первой работы переезжают в последующие. Например, класс Book из «инкапсуляции» становится базовым классом для PrintedBook в теме «Наследование», а затем получает интерфейс Comparable в теме «Полиморфизм».
Документация (единый README.md):

| Лабораторная | Тема                                                                              | Ключевые паттерны/концепции | Ссылка на пакет  |
| :----------- | :-------------------------------------------------------------------------------- | :-------------------------- | :--------------- |
| №1           | Инкапсуляция                                                                      | POJO, Getters/Setters       | ru.yourname.lab1 |
| №2           | Перегрузка                                                                        | Цепочка this(), Фабрика     | ru.yourname.lab2 |
| ...          | ...                                                                               | ...                         | ...              |
| ИИ-анализ    | Отдельный раздел со сравнительной таблицей эффективности Copilot по всем задачам. |                             |                  |
|              |                                                                                   |                             |                  |

### 2. Письменный аналитический отчет (Portfolio Report)

Один PDF-файл или документ Markdown, объединяющий рефлексию по всему блоку.

Сравнительный анализ использования ИИ: Единая таблица, показывающая, сколько времени заняло ручное написание против генерации, количество исправлений после ИИ, типичные ошибки нейросети (логические vs синтаксические).
Архитектурный рост: Описание того, как менялся стиль именования, обработка исключений и комментирование от первой работы к последней.
Разбор ошибок: Список допущенных архитектурных промахов (например, использование изменяемых статических полей без синхронизации) и способы их решения.

### 3. Устная защита или демонстрация проекта

Короткая презентация (5–7 минут) перед преподавателем или комиссией.

Live-кодинг не требуется. Студент открывает финальную версию IDEA.
Демонстрация сценария: Запуск главного класса (например, ModuleRunner.java), который последовательно выполняет тесты всех лабораторных работ, показывая консольный вывод.
Показ различий: Открытие вкладки Git Blame / Git History, чтобы показать авторство строк: какие части написаны вручную (зеленым/своим именем), а какие предложены ИИ (серым/copilot[bot]).
Ответы на вопросы: Умение объяснить выбор конкретного модификатора доступа или обосновать необходимость перегрузки метода именно в контексте бизнес-задачи.

Критерии итоговой оценки блока:

Целостность: Код разных лабораторий не противоречит друг другу стилистически.
Инженерная культура: Использование VCS осмысленно (коммиты атомарны), код покрыт Javadoc.
Критическое мышление при работе с ИИ: Студент может аргументированно доказать, почему он принял или отклонил предложение нейросети.
Автоматизация: Проект собирается одной командой (mvn clean install или через Gradle wrapper) без ручной настройки путей в IDE.
