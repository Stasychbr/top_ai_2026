# Методические указания преподавателям по использованию ресурсов

## Методические указания преподавателям по использованию сквозных предметных областей

### Суть подхода

Вместо абстрактных сущностей (Book, User, Product), которые часто лишены контекста и быстро надоедают студентам, весь семестр строится вокруг одной выбранной предметной области. Студент получает свой вариант в начале курса (например, «Банковское дело») и реализует все лабораторные работы на базе классов этой сферы.

### Что это дает учебному процессу:

#### Рост мотивации через контекст 

Студенту интереснее моделировать реальный банковский счет с проверкой овердрафта или систему умного дома, чем писать десятый по счету класс Book с полями title и author. Это снижает ощущение «программирования ради программирования».

#### Глубокое понимание доменной логики 

 При работе с одним объектом студент вынужден продумывать его жизненный цикл. К десятой лабораторной он знает о своем классе всё: как объект рождается (конструкторы/фабрики), как изменяется (состояние), какие у него есть граничные условия (валидация) и когда он перестает существовать.

#### Естественное усложнение архитектуры. 

Курс Java органично накладывается на развитие реального проекта:
- Лабораторная 1–2: Простые классы, инкапсуляция, конструкторы.
- Лабораторная 3: Наследование (например, от базового BankAccount создаются SavingsAccount и CreditAccount).
- Лабораторная 4: Коллекции счетов (Map<Long, BankAccount> клиентов).
- Лабораторная 5–6: Generics для создания репозитория <T extends Account>, исключения для бизнес-ошибок (InsufficientFundsException).
- Лабораторная 7–8: Сериализация состояния системы в JSON/XML (выгрузка всех транзакций клиента).
- Лабораторная 9–10: Многопоточность при одновременном доступе нескольких операторов к одному счету.

#### Развитие навыков интервьюирования

- На собеседованиях Senior/Middle разработчиков просят проектировать системы именно в конкретных бизнес-доменах (Billing system, Warehouse management). Этот подход готовит студентов к таким задачам заранее.

#### Упрощение проверки для преподавателя

- Преподаватель быстрее погружается в код конкретной группы, так как из года в год видит одни и те же архитектурные паттерны внутри выбранного домена, что позволяет давать более глубокую качественную обратную связь.

## Инструкции по внедрению

- Варианты распределяются случайным образом или по желанию в начале семестра. В группе не должно быть больше 2–3 человек с одинаковым вариантом во избежание прямого копирования бизнес-логики
- Для каждой лабораторной работы преподаватель задает жесткую техническую часть (какие ключевые слова использовать: extends, implements, synchronized), а наполнение полей студенты берут из своего варианта.
- Поощряется добавление собственных полей, релевантных теме. Если тема — многопоточность, студент-банкир должен синхронизировать доступ к балансу, а студент-медик — к очереди пациентов у кабинета.
- Итоговый проект объединяет предыдущие наработки в командный серверный ETL-сервис выбранной предметной области с PostgreSQL и HTTP API.


## Набор заданий по вариантам для лабораторных работ

Номер варианта назначается один раз и сохраняется во всех индивидуальных работах. Для лабораторной №15 отдельный файл вариантов не требуется: студент использует ту же предметную область и основную сущность, что и в работах №1–14. Для командного [мини-проекта №16](../../Project/README.md) участники выбирают один из уже назначенных им вариантов; объединять несколько предметных областей не требуется. Для части вариантов доступны стартовые [датасеты](../../resources/datasets/README.md).
