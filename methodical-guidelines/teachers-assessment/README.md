# Методические указания преподавателям по оцениванию

Данные рекомендации предназначены для унификации оценки практических заданий. Главная цель — проверить не только работоспособность кода, но и понимание студентом фундаментальных принципов объектно-ориентированного программирования (ООП), чистоту архитектуры и умение использовать современные инструменты разработки.

## Критерии общей оценки

**Работоспособность** Программа компилируется без ошибок, проходит базовые тесты на предоставленных данных.
**Соответствие ТЗ** Реализованы все пункты задания в полном объеме.
**Чистота кода** Соблюдение Java Code Conventions (CamelCase для классов/методов, UPPER_SNAKE_CASE для констант). Отсутствие «магических чисел», осмысленные имена переменных (bookList вместо list1), отсутствие закомментированного или мертвого кода.
**Инкапсуляция и архитектура** Поля закрыты модификатором private, логика отделена от представления, отсутствуют глобальные переменные там, где они не нужны.
**Работа с ИИ-инструментом** Студент может объяснить каждую строчку сгенерированного кода, обосновать выбор решения и продемонстрировать сравнение ручного написания с генерацией. Оценка снижается, если код принят «как есть» без проверки типов и бизнес-логики.
**Рефлексия** Наличие краткого комментария к решению (что было сложным, почему выбран именно этот подход).

Для стандартизации проверки используется балльная система (максимум за работу — 100 баллов).

| Категория                             | Макс. балл | Критерии                                                                                                                                                            |
| :------------------------------------ | :--------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Функциональность (ТЗ)                 | 40         | Работают ли все заявленные функции? Корректны ли бизнес-правила (например, расчет бонусов)? Нет ли падения при граничных значениях?                                 |
| Архитектура и чистота кода            | 30         | Правильное применение инкапсуляции/наследования/полиморфизма. Читаемость имен. Отсутствие дублирования логики (DRY). Использование Stream API там, где это уместно. |
| Обработка ошибок                      | 15         | Проверка входных данных (null, пустые строки), использование корректных типов исключений (checked/unchecked), наличие блока finally или try-with-resources.         |
| Использование инструментов            | 10         | Настройка Git (.gitignore, история коммитов), работа со сборщиками (Maven/Gradle), грамотное применение IDEA-фич (Alt+Insert) и ИИ-помощников.                      |
| Производительность и масштабируемость | 5          | Замеры времени выполнения операций (поиск в коллекциях, многопоточность), осознанный выбор структур данных (ArrayList vs HashSet).                                  |

Шкала перевода:

90–100 баллов: Отлично. Код сформирован, рефакторинга не требует, ИИ использован как инструмент усиления.
75–89 баллов: Хорошо. Работает стабильно, есть незначительные огрехи в стиле или неоптимальные циклы.
60–74 балла: Удовлетворительно. Требуется существенный рефакторинг, нарушены принципы ООП, код работает только на «золотом пути».
Менее 60 баллов: Неудовлетворительно. Задание не выполнено или падает на базовых тестах.

## Лабораторная работа №1: Настройка окружения. Первый класс

**Критерии оценки**

- Окружение настроено, JDK 17+ верифицирован.
- Код оформлен согласно Code Style (CamelCase, отступы 4 пробела).
- Поля закрыты (private), присутствуют все пары getter/setter без логических ошибок.
- Метод getDescription() возвращает строку строго в требуемом формате.
- В main созданы три разных объекта, вывод не содержит исключений.
- Проект находится под версионным контролем Git (минимум 2–3 осмысленных коммита).

## Лабораторная работа №2: Конструкторы, перегрузка, static

**Критерии оценки**

- Все пять конструкторов/модификаторов работают корректно, вызовы делегируются через this(...).
- Статический счетчик увеличивается при создании каждого нового объекта (через любой конструктор или фабрику).
- Метод getDescription(boolean) возвращает разные строки в зависимости от значения аргумента.
- Программа компилируется, вывод консоли соответствует ожидаемому формату.
- Код закоммичен в отдельную ветку Git, присутствует сравнение подходов с ИИ.

## Лабораторная работа №3: Наследование и полиморфизм

**Критерии оценки**

- Корректная иерархия (abstract, extends, implements).
- Цепочка конструкторов через this(...) работает без дублирования кода.
- Статический счетчик увеличивается при создании каждого объекта.
- Полиморфизм подтвержден выводом консоли.
Код закоммичен в отдельную ветку Git.
